Docker

What is Docker and why is it useful?

Docker is a tool that lets you package an application with all its dependencies into a container, so it can run the same way on any machine.

⭐ This starts a PostgreSQL server without needing to install anything on your system - just the Docker engine.

docker run -d -p 5432:5432 postgres
How do Dockerfile, Docker image, and container relate to each other?

  • A Dockerfile is a simple text file with instructions for building a Docker image — like which base system to use, what files to copy, and how to run the app.

  • When you build a Dockerfile, it creates a Docker image — a snapshot of the app with everything it needs: code, dependencies, OS files.

  • A container is what you get when you run that image — it's the live, running version of the app, isolated from your system.

How Docker works?

  1. Install Docker – This sets up the Docker engine, a lightweight tool that runs containers on your system.

  2. Build a Docker image – Use a Dockerfile to define your app and its environment.

  3. Run containers from the image – You can start multiple isolated containers from the same image. Also, you can run a Node.js server in one container and a PostgreSQL database in another — all at the same time, for testing your app.

TeamCity

What is TeamCity used for?

TeamCity is used to automate the software delivery process. It can pull code from a repository, build the app, run tests, and deploy the result. It helps ensure that changes are tested and integrated regularly.

How TeamCity works?

  1. TeamCity Server is the central control unit.

    • It manages projects, build configurations, build history, user permissions, and UI.
    • It queues builds, stores logs, and controls agents.

  2. Build Agents are the machines (can be physical, virtual, or Docker containers) that actually run the builds.

    • Each agent has a list of capabilities (e.g., .NET SDK, Node.js, Docker).
    • The server sends a build job to an idle agent that matches the build requirements.

  3. Version Control System (VCS) Integration

    • TeamCity connects to Git, GitHub, Azure DevOps, etc.
    • It checks for changes based on configured VCS roots.
    • A VCS trigger can start a build automatically when new code is pushed.

  4. Build Configuration - each configuration contains:

    • Steps (e.g., npm install, dotnet build, docker build)
    • Triggers (e.g., VCS change, schedule)
    • Parameters (e.g., environment names, secrets)
    • Artifact rules to store outputs

  5. Build Process

    • When triggered, the server selects an agent and sends the build instructions.
    • The agent clones the repo, executes build steps, collects test results, and sends data back to the server.
    • Artifacts (like compiled files or Docker images) are saved for download or further deployment.
